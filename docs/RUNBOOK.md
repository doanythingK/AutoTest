# ERP 매출등록 자동화 운영 런북

## 실행 전 확인

1. Chrome에 ERP 아이디와 비밀번호가 저장되어 있는지 확인한다.
2. Chrome이 원격 디버깅 포트로 실행되어 있는지 확인한다.
   - 기본 포트: `9222`
   - 기본 프로필: `Default`
3. WPF 프로그램의 `Chrome 연결 설정`에서 Chrome 경로, 프로필 디렉터리, 원격 디버깅 포트, 단계 대기 시간을 확인한다.
4. ERP 화면 응답이 느리면 단계 대기 시간을 늘린다.

자동화는 저장된 로그인 정보를 전제로 동작한다. 로그인 필드가 보이면 저장된 로그인 정보 자동 채움만 짧게 기다리며, 아이디나 비밀번호를 새로 입력하거나 변경하지 않는다.

## Chrome 실행

WPF의 `Chrome 실행` 버튼을 사용하면 Chrome을 연 뒤 단계 대기 시간 동안 원격 디버깅 포트 연결 상태를 반복 확인한다. 화면 로그에는 실제 실행 설정과 수동 실행 명령이 함께 표시된다. 직접 실행해야 할 경우 아래 명령을 사용한다.

```powershell
chrome.exe --remote-debugging-port=9222 --profile-directory=Default
```

기존 Chrome이 원격 디버깅 없이 이미 실행 중이면 포트 연결이 실패할 수 있다. 이 경우 Chrome을 모두 종료한 뒤 다시 실행한다.
연결 확인에 실패하면 WPF 실행 로그에 같은 안내와 수동 실행 명령이 함께 표시된다.

## WPF 실행 순서

앱 실행 파일 생성과 개발 실행 명령은 [BUILD_AND_RUN.md](BUILD_AND_RUN.md)를 기준으로 한다.

1. `Chrome 연결 설정`을 확인하고 필요하면 `설정 저장`을 누른다.
2. `Chrome 연결 확인`을 눌러 연결 상태를 확인한다.
3. 수량을 입력한다.
4. 단가를 입력한다.
5. 거래처코드를 입력한다.
6. 계정코드(대변)를 입력한다.
7. 화면에 표시된 예상 공급가액과 예상 세액을 확인한다.
8. 입력값 상태가 `입력값 확인 완료`인지 확인하고 `자동화 실행`을 누른다.
9. 실행 로그에서 `[01/30]`부터 `[30/30]`까지 진행 상황을 확인한다.

거래처코드와 계정코드에는 공백을 넣지 않는다. 앞뒤 공백은 제거되지만, 중간 공백이 있으면 실행할 수 없다.

거래일자는 사용자가 입력하지 않는다. WPF에는 실행일 기준 오늘 날짜가 표시만 되며, ERP 입력칸 형식에 맞춰 `yyyy-MM-dd`, `yyyy.MM.dd`, `yyyyMMdd` 중 하나로 입력된다.

## 자동화 중 지켜야 할 사항

- ERP 아이디와 비밀번호 입력칸은 직접 입력하지 않는다.
- 비밀번호 입력이나 변경을 요구하는 화면이 나오면 자동화로 처리하지 않고 수동으로 확인한 뒤 다시 실행한다.
- 자동화 중 Chrome 탭을 닫지 않는다.
- 자동화 중 ERP 화면을 다른 메뉴로 이동하지 않는다.
- 자동화 실행 중에는 `Chrome 연결 확인`, `Chrome 실행`, `설정 저장` 버튼이 비활성화된다. Chrome 설정 변경이 필요하면 실행을 중지하거나 완료된 뒤 변경한다.
- 저장/전기 중 브라우저 확인창이 뜨면 자동화가 메시지를 로그에 남기고 확인한다.
- 자동화가 멈춘 것처럼 보여도 단계 대기 시간이 끝날 때까지 기다린다.
- 중단이 필요하면 WPF의 `중지` 버튼을 누른다.
- Chrome 연결 확인 중 `중지`를 누른 경우 연결 실패가 아니라 자동화 중지로 기록된다.
- 로그인에 성공한 탭은 자동화 완료 후에도 닫지 않는다.
- 라인저장 전 품목명이 보이고 수량, 단가, 공급가액, 세액이 각 라벨 주변에서 입력값과 맞는지 확인한다.

로그인 성공 화면이 새 탭으로 열리면 프로그램은 열린 탭으로 전환한 뒤 회계관리 단계부터 계속 진행한다.

## 결과 확인

자동화는 다음 항목을 확인한다.

- 계산 결과의 공급가액과 세액
- 라인 목록의 품목, 수량, 단가, 공급가액, 세액
- 거래전기 완료 상태
- 회계전표입력 화면 이동
- 원장전기 완료 상태

회계전표입력 화면이 새 탭으로 열리면 프로그램은 열린 탭으로 전환한 뒤 원장전기 단계를 계속 진행한다.

거래일자, 거래처코드/명, 담당부서, 품목코드/품목명(적요), 계정코드(대변)는 ERP 화면 라벨이 일부 축약되어도 프로그램이 대체 라벨명으로 입력칸을 다시 찾는다.

숫자 검증은 화면 텍스트에서 콤마와 공백을 제거한 뒤 비교한다.

예를 들어 수량 `15,000`, 단가 `275`를 입력하면 공급가액은 `4,125,000`, 세액은 `412,500`으로 확인한다.

## 로그 확인

WPF의 `로그 폴더 열기` 버튼을 누르면 실행 로그 폴더가 열린다. `최근 로그 열기` 버튼은 현재 실행에서 만든 로그가 있으면 그 파일을 열고, 앱을 다시 켠 상태라면 로그 폴더에서 가장 최근 `erp_run_*.log` 파일을 찾아 연다.

```text
%LOCALAPPDATA%\AutoTest.ErpAutomation\RunLogs
```

실행 로그에는 입력값, 예상 공급가액/세액, 실행 PC/사용자, Chrome 설정, 단계별 로그, 최종 실행 결과, 종료 시각, 소요 시간이 저장된다. 파일명은 밀리초 단위로 생성해 빠른 재실행 시에도 기존 로그를 덮어쓰지 않는다.

## 실패 자료 확인

자동화 실패 시 WPF의 `실패 자료 열기` 버튼으로 저장 폴더를 열 수 있다. `최근 실패 HTML`, `최근 실패 화면` 버튼을 누르면 가장 최근에 저장된 실패 HTML 또는 PNG를 바로 연다.

```text
%LOCALAPPDATA%\AutoTest.ErpAutomation\Failures
```

저장되는 파일은 다음과 같다.

- 실패 화면 PNG
- 실패 시점 전체 프레임 HTML

실패 당시 탭 제목, URL, 입력값, 예상 공급가액/세액은 WPF 실행 로그의 실패 지점과 실패 HTML 상단에 함께 기록된다.
실패 HTML에는 각 프레임의 보이는 버튼, 입력칸, 드롭다운 후보와 화면 좌표 요약도 함께 기록된다. 비밀번호나 로그인 정보로 보이는 입력값은 요약과 전체 프레임 HTML에서 마스킹한다.
실패 화면과 HTML은 실제 ERP DOM selector 보강에 사용한다.
PNG와 HTML은 같은 밀리초 단위 파일명 세트로 저장하며, 빠른 재실패 시에도 기존 실패 자료를 덮어쓰지 않는다. 두 파일은 각각 저장하므로 한쪽 저장이 실패해도 가능한 자료는 남긴다.

## 자주 보는 실패와 조치

### Chrome 연결 실패

- Chrome이 원격 디버깅 포트로 실행되었는지 확인한다.
- 포트가 다르면 WPF의 원격 디버깅 포트 값을 맞춘다.
- 기존 Chrome을 모두 종료한 뒤 `Chrome 실행`으로 다시 실행한다.

### 로그인 성공 확인 실패

- 저장된 아이디와 비밀번호가 자동 입력되어 있는지 확인한다.
- 비밀번호 변경이나 추가 인증 화면이 뜬 경우 자동화가 중단된다. 수동으로 처리한 뒤 다시 실행한다.
- 자동화는 아이디/비밀번호 입력칸을 직접 수정하지 않는다.

### 버튼 또는 메뉴 클릭 실패

- 화면에 대상 버튼이 실제로 보이는지 확인한다.
- ERP 화면이 느리면 단계 대기 시간을 늘린다.
- 실패 자료의 PNG/HTML을 확인해 버튼 문구나 라벨이 바뀌었는지 확인한다.

### 계산 결과 확인 실패

- 수량과 단가 입력값이 올바른지 확인한다.
- 자동화는 공급가액/세액이 `0`으로 감지되면 수량과 단가를 재입력하고 계산을 한 번 재시도한다.
- 재시도 후에도 실패하면 실패 자료를 확인한다.

### 라인저장 후 목록 확인 실패

- 품목명 `차피 압축`이 라인 목록에 반영되었는지 확인한다.
- 수량, 단가, 공급가액, 세액이 예상값과 다르면 전기 절차를 계속 진행하지 않는다.
- 실패 자료의 HTML에서 라인 목록 영역 selector가 바뀌었는지 확인한다.
